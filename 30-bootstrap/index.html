Part 6 · Chapter 30

Bootstrap Methodology (Building an Agent Layer by Layer)

The prompts/ directory reveals a systematic methodology for building a coding agent from scratch. This is the most valuable pattern in the entire codebase — it shows how to build an agent in the right order.

30.1 The Layer Dependency Graph

Layer 0: Runtime Shims
    │     (bun:bundle feature flags, MACRO globals)
    ▼
Layer 1: Dependencies & Auth
    │     (package install, API key, env vars)
    ▼
Layer 2: Build System
    │     (esbuild bundler, dev mode)
    ▼
Layer 3: Terminal UI Foundation
    │     (React + Ink renderer)
    ▼
Layer 4: Tool System
    │     (40+ tools, registry, permissions)
    ▼
Layer 5: Command System
    │     (50+ slash commands)
    ▼
Layer 6: Query Engine
    │     (Core LLM loop, streaming, tool calls)
    ▼
Layer 7: System Prompts & Context
    │     (Prompt construction, memory, context)
    ▼
Layer 8: MCP Integration
    │     (External tool/resource protocol)
    ▼
Layer 9: Services Layer
    │     (Analytics, policies, settings)
    ▼
Layer 10: IDE Bridge
    │      (VS Code, JetBrains integration)
    ▼
Layer 11: Testing
          (Smoke tests, integration tests)

30.2 Each Layer Follows This Pattern

1. CONTEXT   — What this layer does and why it exists
2. KEY FILES — What to read and modify
3. TASK      — Broken into lettered parts (A, B, C, D, E, F)
4. VERIFY    — How to confirm it works (testable assertions)

30.3 Example: Layer 4 (Tool System) Build-Out

Part A: Understand the Tool interface
  → Read src/Tool.ts, document the interface

Part B: Audit the tool registry
  → Read src/tools.ts, inventory all tools by category
  → Identify: always-available, feature-gated, internal-only

Part C: Verify each tool compiles
  → Focus on core 10 tools first
  → Check imports, exports, interface compliance

Part D: Fix import issues
  → Feature-gated imports with bun:bundle shim
  → Ant-only tools graceful exclusion

Part E: Create smoke test
  → scripts/test-tools.ts that loads and lists all tools

Part F: Stub internal tools
  → Clean exclusion of Anthropic-internal tools

30.4 The Validation Checkpoint Pattern

Each layer ends with concrete, executable verification:

# Layer 0: Shims
bun -e "import './src/shims/macro.ts'; console.log(MACRO.VERSION)"

# Layer 1: Auth
ANTHROPIC_API_KEY=sk-ant-... bun scripts/test-auth.ts

# Layer 3: UI
bun scripts/test-ink.tsx  # Renders to terminal and exits

# Layer 4: Tools
bun scripts/test-tools.ts  # Lists all 40+ tools

# Layer 6: Query Engine
ANTHROPIC_API_KEY=sk-ant-... bun scripts/test-query.ts "What is 2+2?"

# Layer 7: System Prompts
bun scripts/test-prompt.ts  # Dumps full system prompt

30.5 Service Initialization Pattern (Graceful Degradation)

┌────────────────────────────────────────────────────┐
│  Tier 1: Fire-and-Forget (non-essential)           │
│  ├── GrowthBook (feature flags)                    │
│  ├── Bootstrap data                                │
│  └── Remote settings                               │
│  → Log warning if unavailable, never crash         │
│  → Feature flags fall back to env vars             │
├────────────────────────────────────────────────────┤
│  Tier 2: Lazy-Init (check when needed)             │
│  ├── Policy limits                                  │
│  └── Token estimation API                          │
│  → Stub returns "no limits" if unreachable         │
│  → Fall back to rough estimation                   │
├────────────────────────────────────────────────────┤
│  Tier 3: Local-First (always available)            │
│  ├── Session memory (local filesystem)             │
│  ├── Cost tracking (local display)                 │
│  └── File state cache (in-memory)                  │
│  → No external dependencies                        │
└────────────────────────────────────────────────────┘

Lesson: Build agents layer by layer, bottom up. Each layer must be independently verifiable before the next layer is built on top. This prevents debugging cascading failures from layers you can’t isolate.

See also: Section 37 (Practical Guide) for a concrete implementation starting from Day 1, and Section 22 (Testing) for the corresponding test pyramid.
