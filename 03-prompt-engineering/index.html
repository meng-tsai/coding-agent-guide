System Prompt Engineering

The system prompt is the single most important factor in agent quality. Claude Code constructs its system prompt through a multi-layered composition system with strict priority ordering.

Prompt Composition Architecture

// From src/utils/systemPrompt.ts
function buildEffectiveSystemPrompt(): string {
  // Priority order (highest to lowest):
  // 1. Override prompt (e.g., loop mode) — REPLACES all
  // 2. Coordinator mode prompt — specialized distributed execution
  // 3. Agent system prompt — built-in or custom agent prompts
  //    - In proactive mode: APPENDS to default
  //    - Otherwise: REPLACES default
  // 4. Custom system prompt (--system-prompt flag)
  // 5. Default system prompt — main Claude Code prompt
  // 6. Append prompt — always added last
}

System Prompt Components

The system prompt is assembled from these sections:

┌─────────────────────────────────────────────┐
│ 1. STATIC INSTRUCTIONS                      │
│    Core behavior rules from prompts.ts      │
│    - Role definition                         │
│    - Output style and constraints            │
│    - Safety boundaries                       │
│    - General capabilities                    │
├─────────────────────────────────────────────┤
│ 2. DYNAMIC CONTEXT                          │
│    From context.ts:                          │
│    - Operating system + version              │
│    - Shell type and version                  │
│    - Git repo status, branch, recent commits │
│    - Working directory                       │
│    - Current date                            │
│    - Model name and capabilities             │
├─────────────────────────────────────────────┤
│ 3. TOOL DESCRIPTIONS                        │
│    Auto-generated from tool schemas:         │
│    - Each tool's name and description        │
│    - Input schema (JSON Schema from Zod)     │
│    - Usage instructions from tool.prompt()   │
│    - Permission requirements                 │
├─────────────────────────────────────────────┤
│ 4. MEMORY (CLAUDE.md files)                 │
│    From memdir/:                             │
│    - Project-level CLAUDE.md                 │
│    - User-level ~/.claude/CLAUDE.md          │
│    - Nested memory from subdirectories       │
│    - Session-level extracted memories         │
├─────────────────────────────────────────────┤
│ 5. USER CONTEXT                             │
│    - User settings and preferences           │
│    - Project configuration                   │
│    - Organization policies                   │
│    - Active MCP server instructions          │
│    - Available skill descriptions            │
└─────────────────────────────────────────────┘

Prompt Engineering Principles

3.1 Role Definition and Constraints

The system prompt begins by establishing identity, capabilities, and boundaries:

You are Claude Code, Anthropic's official CLI for Claude.
You are an interactive agent that helps users with software engineering tasks.

Then immediately sets output constraints:

- All text you output outside of tool use is displayed to the user.
- Keep your text output brief and direct.
- Lead with the answer or action, not the reasoning.

Lesson: Role clarity + output constraints dramatically improve agent behavior. Tell the model both what it IS and HOW it should communicate.

3.2 Tool Usage Instructions (Embedded in System Prompt)

Each tool contributes its own section to the system prompt via tool.prompt():

# Using your tools
- Do NOT use Bash to run commands when a relevant dedicated tool is provided.
  - To read files use Read instead of cat, head, tail, or sed
  - To edit files use Edit instead of sed or awk
  - To create files use Write instead of cat with heredoc
  - To search for files use Glob instead of find or ls
  - To search content use Grep instead of grep or rg

Lesson: Explicitly tell the model when NOT to use certain tools. LLMs tend to default to bash for everything unless guided otherwise.

3.3 Safety Instructions

The system prompt includes detailed safety guidance:

# Executing actions with care
Carefully consider the reversibility and blast radius of actions.
For actions that are hard to reverse, affect shared systems, or
could be destructive, check with the user before proceeding.

This is followed by concrete examples:

Examples of risky actions that warrant user confirmation:
- Destructive operations: deleting files/branches, dropping database tables
- Hard-to-reverse operations: force-pushing, git reset --hard
- Actions visible to others: pushing code, creating PRs, sending messages

Lesson: Safety instructions must be specific and example-driven, not abstract. “Be careful” is useless; “don’t force-push without asking” is actionable.

3.4 Context-Aware Prompt Injection

Dynamic context is injected at query time:

// From src/context.ts
async function getSystemContext(): Promise<string> {
  const gitStatus = await getGitStatus()  // branch, status, recent commits
  return [
    `Platform: ${process.platform}`,
    `Shell: ${shell}`,
    `OS Version: ${osVersion}`,
    `Model: ${modelName}`,
    `Working directory: ${cwd}`,
    `Git: ${gitStatus}`,
    `Date: ${currentDate}`,
  ].join('\n')
}

Lesson: Dynamic context should be computed at query time, not session start. Git status changes between queries.

3.5 Memory Injection

CLAUDE.md files are injected as system context:

// Memory hierarchy
Project memory:  ./CLAUDE.md           → Project conventions
User memory:     ~/.claude/CLAUDE.md    → Cross-project preferences
Nested memory:   ./src/CLAUDE.md        → Subdirectory-specific context

Lesson: Hierarchical memory allows context to be scoped. Project-level memory doesn’t pollute other projects; user-level memory carries across all projects.

3.6 Real System Prompt Patterns (From the Actual Source)

These are actual patterns extracted from src/constants/prompts.ts:

Pattern 1: Identity + Role + Constraints (Opening)

You are Claude Code, Anthropic's official CLI for Claude.
You are an interactive agent that helps users with software engineering tasks.
Use the instructions below and the tools available to you to assist the user.

Three sentences that establish: who you are, what you do, how you do it. No fluff.

Pattern 2: Negative Constraints (What NOT To Do)

The real prompt devotes more space to negative constraints than positive ones:

- Don't add features, refactor code, or make "improvements" beyond what was asked.
  A bug fix doesn't need surrounding code cleaned up.
- Don't add error handling, fallbacks, or validation for scenarios that can't happen.
  Trust internal code and framework guarantees.
- Don't create helpers, utilities, or abstractions for one-time operations.
  Three similar lines of code is better than a premature abstraction.
- Don't add docstrings, comments, or type annotations to code you didn't change.
  Only add comments where the logic isn't self-evident.

Lesson: Negative constraints prevent the most common failure modes. LLMs naturally over-engineer and add unsolicited improvements. Tell them to stop.

Pattern 3: Concrete Examples Over Abstract Rules

Instead of “be careful with destructive actions”, the real prompt gives specific examples:

Examples of risky actions that warrant user confirmation:
- Destructive operations: deleting files/branches, dropping database tables,
  killing processes, rm -rf, overwriting uncommitted changes
- Hard-to-reverse operations: force-pushing, git reset --hard, amending
  published commits, removing or downgrading packages
- Actions visible to others: pushing code, creating/closing/commenting on PRs
  or issues, sending messages (Slack, email, GitHub)

Pattern 4: Tool Routing Instructions

Do NOT use the Bash to run commands when a relevant dedicated tool is provided:
  - To read files use Read instead of cat, head, tail, or sed
  - To edit files use Edit instead of sed or awk
  - To create files use Write instead of cat with heredoc or echo redirection
  - To search for files use Glob instead of find or ls
  - To search the content of files, use Grep instead of grep or rg

Lesson: Without explicit routing, the model defaults to bash for everything. Explicit tool routing dramatically improves tool selection accuracy.

Pattern 5: Output Style With Specific Guidance

Keep your text output brief and direct. Lead with the answer or action,
not the reasoning. Skip filler words, preamble, and unnecessary transitions.
Do not restate what the user said — just do it.

Focus text output on:
- Decisions that need the user's input
- High-level status updates at natural milestones
- Errors or blockers that change the plan

If you can say it in one sentence, don't use three.

3.7 Specialized Agent Prompt Patterns

Claude Code defines 5 built-in agents, each with a carefully crafted system prompt. These demonstrate how to specialize agent behavior through prompting alone.

The Verification Agent (Adversarial Testing)

The verification agent’s prompt is perhaps the most instructive. It explicitly defines two failure modes to avoid:

You are a verification specialist. Your job is not to confirm the
implementation works — it's to try to break it.

You have two documented failure patterns:
1. Verification avoidance — skipping tests or accepting "looks right"
2. Being seduced by the first 80% — finding the obvious cases work
   and declaring success without testing edge cases

Your entire value is in finding the last 20%.

It then prescribes an adversarial methodology:

Adversarial probes:
- What happens with empty input?
- What happens with maximum-size input?
- What happens when a dependency is unavailable?
- What happens during concurrent access?
- What happens if the user is not authenticated?

Lesson: Agent specialization through prompting is extremely effective. You don’t need different models or architectures — a well-crafted system prompt creates genuinely different behavior.

The Explore Agent (Read-Only Constraint)

=== CRITICAL: READ-ONLY MODE - NO FILE MODIFICATIONS ===
[Lists all prohibited operations]

Your role is EXCLUSIVELY to search and analyze existing code.
You do NOT have access to file editing tools.

This uses both system prompt constraints AND tool restrictions (the agent only receives read-only tools). Belt and suspenders.

The Coordinator Agent (Synthesis-First)

You are a coordinator. Your job is to:
- Help the user achieve their goal
- Direct workers to research, implement and verify code changes
- Synthesize results and communicate with the user
- Answer questions directly when possible — don't delegate work
  that you can handle without tools

The key insight: “don’t delegate work you can handle without tools.” Without this instruction, coordinators tend to delegate everything, even simple questions.

Anti-Patterns in Prompt Engineering

  1. Don’t repeat yourself — Say it once, clearly, in the right section
  2. Don’t be vague — “Be helpful” is worthless; “Lead with the answer or action” is actionable
  3. Don’t overload with instructions — Claude Code’s system prompt is long but highly structured with sections
  4. Don’t forget to tell the model what NOT to do — Negative constraints are often more important than positive ones
  5. Don’t use abstract rules — “Be careful” is useless; give concrete examples of what “careful” means
  6. Don’t assume tool routing — Explicitly tell the model which tool to use for which task
