Part 1 · Chapter 4

Context Management

Context management is the art of deciding what information the model sees, when, and in what order. This is arguably the most important engineering challenge in building a coding agent.

The Context Budget

Every conversation has a fixed context window. Claude Code manages this through multiple strategies:

┌──────────────────────────────────────────────┐
│            Context Window Budget              │
│                                              │
│  System Prompt          ~5-15K tokens        │
│  CLAUDE.md memories     ~1-5K tokens         │
│  Conversation history   ~variable            │
│  Tool results           ~variable            │
│  Dynamic context        ~1-2K tokens         │
│  Available tools        ~5-10K tokens        │
│                                              │
│  Total available: ~200K tokens               │
│  (varies by model)                           │
└──────────────────────────────────────────────┘

4.1 File State Cache (Read Deduplication)

When the model reads a file, the content is cached:

// From ToolUseContext
readFileState: FileStateCache  // LRU cache of file reads

// In FileReadTool:
async call(input, { readFileState }) {
  const content = await readFile(absolutePath)

  // Cache for subsequent operations
  readFileState.set(absolutePath, {
    content,
    timestamp: Date.now(),
    offset,
    limit
  })

  return { data: { content, size, mimeType } }
}

Lesson: Cache file reads within a turn. If the model reads a file and then edits it, you don’t need to read it again — the cache has the content. This also prevents edit-after-modification races.

4.2 Tool Result Budgeting

Large tool results can blow the context window. Claude Code uses a result budget system:

// contentReplacementState tracks tool result sizes
// maxResultSizeChars limits individual tool output
const FileReadTool = buildTool({
  maxResultSizeChars: Infinity,  // Never persist — would create recursive reads
})

const BashTool = buildTool({
  maxResultSizeChars: 30_000,  // Truncate bash output at 30K chars
})

Lesson: Set appropriate result size limits per tool. Shell output can be enormous; file reads can be large but are self-referencing (the model needs the full content to edit accurately). Different tools need different limits.

4.3 Tool Deferral (Lazy Tool Loading)

Not all tools need to be in the system prompt at all times. Deferred tools are hidden until explicitly needed:

const WebSearchTool = buildTool({
  shouldDefer: true,  // Hidden initially
})

// ToolSearchTool lets the model discover deferred tools on demand
const ToolSearchTool = buildTool({
  // Matches deferred tool names against a query
  // Returns matching tool schemas
})

How it works:

  1. At prompt time, deferred tools are omitted from the tool list
  2. Their names are mentioned in a system reminder
  3. The model can use ToolSearch to fetch their full schemas
  4. Once fetched, the tool becomes callable

Lesson: Don’t burden the context window with tools the model rarely needs. Let it discover them on demand. This keeps the base prompt smaller and tool selection cleaner.

4.4 Nested Memory Triggering

When the model reads a file in a subdirectory that has its own CLAUDE.md, that memory is injected as a system message:

// nestedMemoryAttachmentTriggers: Set<string>
// loadedNestedMemoryPaths: Set<string>

// When FileReadTool reads from src/components/:
// 1. Check if src/components/CLAUDE.md exists
// 2. If yes, and not already loaded, inject as system message
// 3. Mark as loaded to prevent duplicate injection

Lesson: Context should be loaded lazily based on what the model is actually working on. Don’t front-load all possible context; load it as the model navigates the codebase.

4.5 System Reminders

Claude Code uses <system-reminder> tags to inject context mid-conversation without taking a full turn:

<system-reminder>
The following MCP servers have provided instructions:
### server-name
Use this server for X, Y, Z...
</system-reminder>

Lesson: System reminders are a powerful pattern for injecting context that the model needs to be aware of but that doesn’t warrant a separate message turn.
