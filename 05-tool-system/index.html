Tool System Design

The tool system is Claude Code’s most distinctive architectural feature. Every capability the LLM can invoke — from reading files to spawning sub-agents — is a self-contained tool.

5.1 The Tool Interface

Every tool conforms to this interface:

type Tool<Input, Output, Progress> = {
  // === Identity ===
  name: string
  aliases?: string[]
  searchHint?: string

  // === Execution ===
  call(args, context, canUseTool, parentMessage, onProgress?): Promise<ToolResult<Output>>

  // === Schema & Validation ===
  readonly inputSchema: Input            // Zod schema
  readonly inputJSONSchema?: object      // JSON Schema (for API)
  outputSchema?: z.ZodType<unknown>
  validateInput?(input, context): Promise<ValidationResult>

  // === Prompt Integration ===
  description(input, options): Promise<string>   // Short description for API
  prompt(options): Promise<string>               // System prompt contribution

  // === Permission & Safety ===
  checkPermissions(input, context): Promise<PermissionResult>
  isReadOnly(input): boolean
  isDestructive?(input): boolean
  isConcurrencySafe(input): boolean

  // === UI Rendering ===
  renderToolUseMessage(input, options): React.ReactNode
  renderToolResultMessage?(content, options): React.ReactNode
  renderToolUseErrorMessage?(result, options): React.ReactNode

  // === Metadata ===
  maxResultSizeChars: number
  shouldDefer?: boolean      // Hidden until discovered via ToolSearch
  alwaysLoad?: boolean       // Always in prompt, never deferred
  strict?: boolean           // Strict schema validation
  isMcp?: boolean           // MCP-sourced tool
}

5.2 The buildTool() Factory

Tools are created with a factory that provides safe defaults:

function buildTool<D extends ToolDef>(def: D): Tool {
  return {
    // Fail-closed defaults
    isEnabled: () => true,
    isConcurrencySafe: () => false,    // NOT safe by default
    isReadOnly: () => false,            // NOT read-only by default
    isDestructive: () => false,
    checkPermissions: () => ({ behavior: 'allow' }),
    userFacingName: () => def.name,

    // Merge with provided definition
    ...def,
  }
}

Critical Design Decision: isConcurrencySafe defaults to false. This means new tools are automatically serialized unless explicitly marked as safe. This is the fail-closed principle — unknown tools are treated as potentially dangerous.

5.3 Tool Directory Structure

Each tool is a self-contained module:

src/tools/FileEditTool/
├── FileEditTool.ts      # Main implementation (call, validate, permissions)
├── UI.tsx               # Terminal rendering components
├── prompt.ts            # System prompt contribution text
└── utils.ts             # Tool-specific helpers

Lesson: Self-contained tool modules make it easy to add, remove, or modify tools without affecting others. Each tool owns its own UI, validation, and prompt text.

5.4 Example: FileEditTool (Write Tool, Not Concurrency-Safe)

export const FileEditTool = buildTool({
  name: 'Edit',
  strict: true,
  maxResultSizeChars: 100_000,

  // Safety declarations
  isConcurrencySafe() { return false },  // File writes must serialize
  isReadOnly() { return false },
  isDestructive() { return true },

  // Input validation with race condition detection
  async validateInput(input, context) {
    const { file_path, old_string, new_string } = input
    const absolutePath = expandPath(file_path)

    // Check for edit-after-modification race
    const readTimestamp = readFileState.get(absolutePath)
    if (readTimestamp && getFileModTime(absolutePath) > readTimestamp.timestamp) {
      return {
        result: false,
        behavior: 'ask',  // Ask user before proceeding
        message: 'File modified since last read'
      }
    }

    // Verify the old_string exists in the file
    const content = readFileSync(absolutePath)
    if (!content.includes(old_string)) {
      return {
        result: false,
        message: 'String to replace not found in file'
      }
    }

    return { result: true }
  },

  // Atomic read-modify-write
  async call(input, { readFileState }) {
    const content = readFileSync(absolutePath)
    const newContent = content.replace(old_string, new_string)

    writeFileSync(absolutePath, newContent)

    // Update cache to prevent stale reads
    readFileState.set(absolutePath, {
      content: newContent,
      timestamp: Date.now()
    })

    return { data: { edited: true } }
  }
})

Key Patterns:

  • Race condition detection: Checks if the file was modified between the read and edit
  • Atomic operations: No yields between read and write
  • Cache invalidation: Updates the file state cache after writing

5.5 Example: GlobTool (Read-Only, Concurrency-Safe)

export const GlobTool = buildTool({
  name: 'Glob',
  searchHint: 'find files by name pattern',

  isConcurrencySafe() { return true },   // Safe to run in parallel
  isReadOnly() { return true },

  isSearchOrReadCommand() {
    return { isSearch: true, isRead: false }
  },

  async call(input, { globLimits }) {
    const limit = globLimits?.maxResults ?? 100
    const { files, truncated } = await glob(input.pattern, input.path, { limit })
    return {
      data: { filenames: files.map(toRelativePath), numFiles: files.length, truncated }
    }
  }
})

5.6 Example: BashTool (Complex, Generator-Based)

export const BashTool = buildTool({
  name: 'Bash',
  strict: true,
  maxResultSizeChars: 30_000,

  // Only concurrency-safe if read-only
  isConcurrencySafe(input) {
    return this.isReadOnly?.(input) ?? false
  },

  // Parse compound commands for granular permission matching
  async preparePermissionMatcher({ command }) {
    const parsed = await parseForSecurity(command)
    const subcommands = parsed.commands.map(c => c.argv.join(' '))
    return pattern => subcommands.some(cmd => matchWildcard(pattern, cmd))
  },

  // Generator-based execution with streaming progress
  async call(input, context, _, parentMessage, onProgress) {
    const generator = runShellCommand({ input, abortController: context.abortController })

    let counter = 0
    let result
    do {
      result = await generator.next()
      if (!result.done && onProgress) {
        onProgress({
          toolUseID: `bash-progress-${counter++}`,
          data: {
            type: 'bash_progress',
            output: result.value.output,
            elapsedTimeSeconds: result.value.elapsedTimeSeconds
          }
        })
      }
    } while (!result.done)

    return { data: { stdout: result.value.stdout, stderr: result.value.stderr } }
  }
})

Key Patterns:

  • Generator-based execution: Long-running commands emit progress events via generators
  • Compound command parsing: git push && npm publish is parsed into subcommands for granular permission matching
  • Dynamic concurrency safety: Read-only bash commands (like git status) can run in parallel; write commands cannot

5.7 Tool Registration and Filtering

// src/tools.ts — Single source of truth

function getAllBaseTools(): Tools {
  return [
    AgentTool,
    BashTool,
    FileReadTool,
    FileEditTool,
    FileWriteTool,
    GlobTool,
    GrepTool,
    WebFetchTool,
    WebSearchTool,
    // Feature-gated tools
    ...(feature('WORKFLOW_SCRIPTS') ? [WorkflowTool] : []),
    ...(REPLTool ? [REPLTool] : []),
    // MCP tools added dynamically
    ...(isToolSearchEnabled() ? [ToolSearchTool] : []),
  ]
}

// Filter by permissions
function getTools(permissionContext: ToolPermissionContext): Tools {
  return filterToolsByDenyRules(getAllBaseTools(), permissionContext)
    .filter(tool => tool.isEnabled())
}

// Assemble final pool: built-in + MCP
function assembleToolPool(permissionContext, mcpTools): Tools {
  const builtIn = getTools(permissionContext)
  const allowed = filterToolsByDenyRules(mcpTools, permissionContext)

  // Sort for prompt-cache stability (deterministic ordering)
  return uniqBy(
    [...builtIn].sort(byName).concat(allowed.sort(byName)),
    'name'
  )
}

Lesson: Sort tools deterministically for prompt cache stability. If the tool list changes order between queries, the cached prompt prefix is invalidated.

5.8 Semantic Input Coercion

Tools use semantic helpers to accept user-friendly inputs:

// Input schema with semantic coercion
const inputSchema = z.object({
  timeout: semanticNumber(z.number().optional()),
  // Accepts: "30m", "5 seconds", 30000 → all become 30000ms

  verbose: semanticBoolean(z.boolean().optional()),
  // Accepts: "yes", "true", 1 → all become true
})

Lesson: The LLM sometimes outputs values in unexpected formats. Semantic coercion handles this gracefully.
